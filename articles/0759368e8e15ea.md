## はじめに

AWSのEC2インスタンスに接続する際に、SSHではなくSSM/セッションマネージャーを介して接続する場合は多くあるかと思います。
わたしの勤務する会社でも、以前は秘密鍵を利用したSSH接続を利用していましたが、鍵管理の観点や接続ログ監視が可能という点から、SSMを利用した接続に切り替えました。SSMを利用することでSSH用のポート番号を開ける必要もなくなり、よりセキュアな利用が可能になっています。
（SSMに関しては[このへんとか](https://dev.classmethod.jp/articles/ec2-access-with-session-manager/)がわかりやすかったです）

切り替え直後、たまたまSCP接続でEC2インスタンスにローカルからファイル転送したいタイミングがあり、ちょっと詰まったのでシェアしたいと思います。

## 前提

以下の設定が実施されているものとします。

- ローカルPCにAWS CLIがすでに導入済みであること
- ローカルPCにSSMプラグイン、EC2インスタンスにSSMエージェントが導入済みであり、SSMを介した接続ができていること

## 手順

1. SSH設定の準備

SSMを経由した接続では、通常のSSHのように直接インスタンスに接続するのではなく、
SSMのセッションをトンネルとして利用しています。（以下が詳しいです）
https://baresupport.jp/blog/2022/03/14/82/

そのため、SCPであっても内部的にはSSH接続として扱われるように、
ProxyCommandを設定してあげる必要があります。

基本的に、SSH接続時にはホームディレクトリ配下の`~/.ssh/config`（拡張子なし）を読みに行きます。
この`~/.ssh/config`を以下のように、SSMを経由するようPROXY COMMANDを設定します。

```vim
host i-* mi-*
    ProxyCommand sh -c "aws ssm start-session --target %h --document-name AWS-StartSSHSession --parameters 'portNumber=%p'"
```

ちなみにこの後の手順でコケる場合、aws.exeファイルの絶対パスを`where.exe aws`などで取得し、指定したほうが良いです。（特にwindows）

```vim
host i-* mi-*
    ProxyCommand <aws.exeのフルパス> ssm start-session --target %h --document-name AWS-StartSSHSession --parameters 'portNumber=%p'
```

もし過去にSSHで接続したことがなく、`~/.ssh`ディレクトリが見つからない場合は自分で作成すれば大丈夫です。

```bash
mkdir ~/.ssh
cd ~/.ssh
touch config
vi config
```

2. 鍵の準備

もし鍵がなければ、以下の辺りの解説を参考に秘密鍵を入手してください。
https://business.ntt-east.co.jp/content/cloudsolution/column-513.html

入手した鍵は、先ほど準備した`~/.ssh`ディレクトリ配下に保存すると良いと思います。
保存したら、先ほどの`~/.ssh/config`に以下のように追加しておくと、のちのち楽です。

```vim
host i-* mi-*
    ProxyCommand sh -c "aws ssm start-session --target %h --document-name AWS-StartSSHSession --parameters 'portNumber=%p'"
    IdentityFile {秘密鍵のパス}
    User {EC2インスタンスのユーザー 例：ec2-user}
```

一時的な鍵を利用する方法もあるようなのですが、アタッチされている権限でできるか怪しかったこともあり、もともと持っていた鍵を利用しました。
https://docs.aws.amazon.com/ja_jp/ec2-instance-connect/latest/APIReference/API_SendSSHPublicKey.html

3. SCP接続・ファイル送信

以上の設定ができたら、SCPコマンドで接続、ファイル送信できるはずです。

```bash
scp {local_file_path} i-{instance_id}:/home/{instance_path}
```

ここで鍵の指定をしたければこんな感じになるかと。
`~/.ssh/config`の内容はオーバーライドされます。

```bash
scp -i {key_path} {local_file_path} i-{instance_id}:/home/{instance_path}
```


## トラブルシューティイング

#### ホームディレクトリがわからん

黙って`$USERPROFILE`を探しましょう。

```bash
echo $USERPROFILE
```

#### ssh: Could not resolve hostname... って怒られる

SSH設定ファイルが正しく読み込まれているか確認してください。下記で見れます。

```bash
ssh -G i-{instance_id}
```

#### posix_spawnp: No such file or directory って怒られる

awsコマンドが見つかってないかもしれません。`~/.ssh/config`で`aws.exe`を絶対パスで指定してあげたほうが良いかもです。

#### Recieved message too long　って怒られる

EC2ログイン時に履歴などを`echo`する設定など、SCP接続（非対話モード）の時に何かしら返される設定になっているとエラーになるようです。
わたしは別の方法で`echo`を無効化しましたが、Gemini先生曰くEC2側の`.bashrc`ファイルの編集でなんとかなる場合もあるみたいです。参考まで。

```vi
# 非対話モードの時は処理を終了する
[[ $- != *i* ]] && return
```

## おわりに

シンプルにSCPでファイル転送したいだけでしたが、SSH接続に必要な鍵の知識、EC2に接続する経路を理解するよいタイミングになったように思います。
SSMとSSH/SCP、うまく使い分けられることが一番大事ですね！
